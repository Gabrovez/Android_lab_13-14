# UnscrambleGame

Игра-головоломка, где нужно угадывать слова, составленные из перемешанных букв.

## Описание
Приложение предлагает игроку слово с перемешанными буквами. Задача пользователя — ввести правильное слово в исходном виде. Игра состоит из 10 раундов. За каждый правильный ответ начисляются очки. В конце игры отображается итоговый счет с возможностью начать заново.

## Технологии
*   Kotlin
*   Jetpack Compose
*   ViewModel
*   StateFlow

## Функциональность
*   Отображение перемешанного слова.
*   Поле ввода для ответа пользователя.
*   Проверка правильности введенного слова.
*   Подсчет очков и отображение текущего прогресса (номер слова).
*   Обработка ошибок при неправильном вводе.
*   Возможность пропустить слово без потери очков.
*   Диалог завершения игры с итоговым результатом.
*   Кнопка перезапуска игры.
*   Сохранение состояния игры при повороте экрана.

## Контрольные вопросы

**1. Что такое ViewModel и зачем он нужен?**
ViewModel — это компонент архитектуры Android, предназначенный для хранения и управления данными, связанными с пользовательским интерфейсом, с учетом жизненного цикла. Он решает проблему потери данных при изменении конфигурации (например, повороте экрана). Без ViewModel данные, хранящиеся непосредственно в Activity или Composable-функциях, уничтожаются вместе с ними при пересоздании, что приводит к сбросу состояния интерфейса.

**2. В чём разница между MutableStateFlow и StateFlow?**
`MutableStateFlow` позволяет изменять значение потока изнутри класса (в данном случае из ViewModel), тогда как `StateFlow` предоставляет доступ только для чтения.
Поля `_uiState` и `uiState` создаются для реализации инкапсуляции. `_uiState` (приватное) используется внутри ViewModel для обновления данных, а `uiState` (публичное) передается в UI для наблюдения.
“Backing property pattern” (шаблон резервного свойства) — это практика использования приватного изменяемого свойства и публичного неизменяемого свойства, которое возвращает представление приватного. Это предотвращает случайное изменение состояния извне ViewModel.

**3. Почему в ViewModel нельзя хранить Context или View?**
ViewModel живет дольше, чем Activity или Fragment. Если сохранить ссылку на Context или View в ViewModel, это приведет к утечке памяти, так как система не сможет уничтожить Activity даже после его закрытия, пока жива ViewModel. ViewModel должен содержать только бизнес-логику и данные UI, не связанные с конкретными элементами интерфейса или контекстом приложения.

**4. Чем StateFlow отличается от remember и rememberSaveable?**
`remember` хранит состояние только в памяти текущей композиции и теряется при рекомпозиции или изменении конфигурации. `rememberSaveable` сохраняет состояние через Bundle при изменении конфигурации и восстановлении процесса, но ограничен типами данных, поддерживаемых Bundle.
`StateFlow` используется внутри ViewModel и переживает изменения конфигурации, так как ViewModel сохраняется системой. Однако StateFlow не сохраняется при полной смерти процесса (process death).
`remember` лучше использовать для локального состояния UI внутри одной функции. `rememberSaveable` — для простого сохранения примитивных данных при повороте экрана без сложной логики. `StateFlow` в ViewModel — для управления состоянием экрана, требующим бизнес-логики и устойчивости к изменению конфигурации.

**5. Что такое data class в Kotlin и зачем он используется для UI State?**
`data class` — это класс, предназначенный преимущественно для хранения данных. Для него автоматически генерируются методы `equals()`, `hashCode()`, `toString()` и `copy()`.
Он используется для UI State, чтобы объединить все параметры состояния экрана в один объект. Это упрощает передачу данных, сравнение состояний и обновление отдельных полей.
Функция `copy()` позволяет создать новую копию объекта с измененными значениями некоторых полей, оставляя остальные без изменений, что удобно для иммутабельного обновления состояния.

**6. Как работает collectAsState() в Compose?**
`collectAsState()` — это функция расширения, которая подписывается на поток `StateFlow` и преобразует его в `State<T>`, понятный Compose.
UI автоматически обновляется, потому что Compose отслеживает чтение значений `State`. Когда `StateFlow` emits новое значение, `collectAsState()` обновляет свое внутреннее состояние, что вызывает рекомпозицию функций, использующих это значение.

**7. Зачем разделять код на пакеты data и ui?**
Разделение кода на пакеты (например, `data` для моделей и `ui` для интерфейса и ViewModel) поддерживает принципы чистой архитектуры и разделения ответственности (Separation of Concerns). Это делает код более читаемым, поддерживаемым и тестируемым, так как логика данных отделена от логики отображения.

**8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?**
При полной остановке процесса приложения (process death) экземпляр ViewModel уничтожается, и все данные в нем теряются.
Данные в этом случае не сохраняются.
Для долговременного хранения данных необходимо использовать другие механизмы, такие как Room Database, DataStore или SharedPreferences, которые сохраняют данные на диск.
